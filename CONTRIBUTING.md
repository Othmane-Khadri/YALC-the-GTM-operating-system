# Contributing to YALC

Thanks for your interest in contributing to YALC. Here's how to get started.

## Development Setup

```bash
git clone https://github.com/Othmane-Khadri/YALC-the-GTM-operating-system.git
cd YALC-the-GTM-operating-system
pnpm install
cp .env.example .env.local   # Add at least ANTHROPIC_API_KEY
pnpm typecheck                # Verify everything compiles
pnpm test                     # Run the test suite
```

## Architecture Rules

YALC follows a strict **three-layer pattern**:

```
Service (API wrapper) → Provider (StepExecutor) → Skill (user-facing operation)
```

Never skip layers. A skill should never call an API directly — it goes through a provider, which goes through a service.

## Code Conventions

- **TypeScript** with strict mode, ESM only (`"type": "module"`)
- **File names:** kebab-case (`campaign-manager.ts`)
- **Imports:** Use `.js` extensions in relative imports (ESM requirement) and `node:` prefix for built-ins
- **Tests:** Vitest, co-located in `__tests__/` directories, in-memory SQLite

## Before Submitting a PR

1. `pnpm typecheck` passes with no errors
2. `pnpm test` — all tests pass
3. Any command that sends or writes supports `--dry-run`
4. No API keys or secrets in code — use `sk-...redacted` pattern in examples
5. Campaign outcomes are wired to the intelligence store

## Adding a New Provider

1. Create `src/lib/providers/builtin/<name>-provider.ts`
2. Implement the `StepExecutor` interface from `src/lib/providers/types.ts`
3. Register it in `src/lib/providers/builtin/index.ts`
4. Add required env vars to `.env.example`
5. Write tests in `src/lib/providers/builtin/__tests__/<name>.test.ts`
6. Document the provider in `docs/providers.md`

## Skills

YALC has **three distinct skill runtime systems**. They look similar from the outside (markdown files with frontmatter) but execute on different runtimes for different audiences. Pick the right one before you start authoring.

### The three systems

| Path | Runtime | Audience | Activation |
|------|---------|----------|------------|
| `.claude/skills/<slug>/SKILL.md` | Claude Code conversational skills | YALC users in a Claude Code session | Natural-language phrases match the `description` frontmatter |
| `configs/skills/<slug>.md` | YALC framework-runner steps | The framework runner, declared in `configs/frameworks/*.yaml` | Listed in a framework's `steps:` block |
| `~/.gtm-os/skills/<slug>.md` | User-installed skills via the `skills:create` wizard | One specific user's local install | Loaded at YALC startup from the user's home dir |

**`.claude/skills/`** are conversational entry points. The user types something like "qualify these leads" and Claude Code matches the trigger phrase, reads the SKILL.md, and walks the user through the flow. Usually shells out to `npx tsx src/cli/index.ts <command>`. Reference: `.claude/skills/provider-builder/SKILL.md`.

**`configs/skills/`** are framework-runner steps. Each file has `capability:` and `output_schema:` frontmatter and is invoked by name in a framework yaml's `steps:` list (see `configs/frameworks/outreach-campaign-builder.yaml` for an example). They never activate from a chat phrase. Reference: `configs/skills/outbound-hypothesis-capture.md`.

**`~/.gtm-os/skills/`** are user-authored skills generated by the interactive `yalc-gtm skills:create` wizard. They live outside the repo and ship as part of the user's local install, not the codebase.

### Slug naming

CLI commands use colon-namespaced slugs (`area:command`). Skill folders / files use natural-language `verb-noun` slugs. The two are deliberately different so each surface reads naturally for its consumer.

| CLI | `.claude/skills/` slug | `configs/skills/` slug |
|-----|------------------------|------------------------|
| `leads:qualify` | `qualify-leads` | `qualify-engagers` |
| `leads:scrape-post` | `scrape-post-engagers` | `scrape-post-engagers` |
| `routine:propose` | `build-routine` | n/a |

`yalc.ai/skills/<slug>/` URLs match the `.claude/skills/` slug.

### When to add a skill

Add a `.claude/skills/` skill when **a user wants to talk to YALC about X** in a Claude Code session and X is a high-value flow they would otherwise have to memorize CLI commands for.

Add a `configs/skills/` step when a framework needs a reusable, schema-typed reasoning step (LLM-driven enrichment, classification, scoring) that the runner can wire into multiple frameworks.

Add to `~/.gtm-os/skills/` only via the `skills:create` wizard. Do not hand-author files in that path; they are user-state, not codebase.

### Authoring a `.claude/skills/` skill

1. Create `.claude/skills/<slug>/SKILL.md` with frontmatter:
   - `name:` matches the folder slug
   - `description:` includes a bulleted list of trigger phrases (each phrase is a substring the user might say)
   - `version: 1.0.0`
2. Body: greeting → input gathering → local validation → CLI shell-out (`npx tsx src/cli/index.ts <command>`) → parse JSON → render result.
3. For pure / read-only commands, prefer the import-direct pattern with shell-out fallback. See `docs/skills-architecture.md` for the runtime decision tree.
4. Run `node scripts/lint-skill-triggers.mjs` to catch trigger-phrase substring overlap with sibling skills before opening a PR. The lint check is enforced in CI.
5. Add the skill to the README "Bundled skills" table.

Reference walkthrough: `.claude/skills/provider-builder/SKILL.md` (~230 lines, gold-standard shell-out pattern).

### Authoring a `configs/skills/` step

1. Create `configs/skills/<slug>.md` with frontmatter that matches the framework-runner contract: `name`, `description`, `category`, `inputs[]`, `capability`, `capabilities[]`, `output`, `output_schema`. See `configs/skills/outbound-hypothesis-capture.md` for a minimal example.
2. The body of the file is the LLM prompt that runs at that step. Use `{{input_name}}` placeholders to interpolate the framework runner's input bindings.
3. Wire the step into a framework yaml under `configs/frameworks/`.

### Meta-skills

Some `.claude/skills/` skills (e.g. `prospect-discovery-pipeline`) are **conversational orchestrators** that chain multiple lower-level skills and CLI commands. They live in `.claude/skills/` because activation is conversational, but they are not the smallest unit of work. Mark them clearly in the body so authors don't confuse them with a framework-runner step or a single-CLI wrapper.

### Legacy: TypeScript skill registry

There is also an older `src/lib/skills/builtin/` TypeScript registry (loaded at runtime by `orchestrate` and surfaced via `skills:list`). New work should prefer `.claude/skills/` for conversational flows and `configs/skills/` for framework steps. Only add to `src/lib/skills/builtin/` if you are extending the legacy registry path:

1. Create `src/lib/skills/builtin/<name>.ts`
2. Export a `Skill` object with id, name, description, category, inputSchema, and execute function
3. Register it in `src/lib/skills/builtin/index.ts`
4. Add the skill to `docs/skills.md`

## Reporting Bugs

Open an issue at https://github.com/Othmane-Khadri/YALC-the-GTM-operating-system/issues with:
- What you expected to happen
- What actually happened
- Output of `yalc-gtm doctor`
- Your Node.js version (`node --version`)

## Security Vulnerabilities

Do **not** file public issues for security vulnerabilities. See [SECURITY.md](SECURITY.md) for responsible disclosure instructions.
